# Issue とラベル運用

KeruLabs では、Issue を「何が起きているか」と「誰が着手しやすいか」がすぐ分かる状態で管理します。このドキュメントは、公開リポジトリ移行後もそのまま再利用できる最小ルールです。

## 基本ラベル

| ラベル | 用途 | 付ける条件 |
|--------|------|------------|
| `bug` | 不具合報告 | 期待動作と実際の動作がずれている |
| `enhancement` | 機能改善 | 既存機能の改善や新しい体験の提案 |
| `documentation` | 文書改善 | README、docs、コメント、チュートリアル改善 |
| `question` | 質問 | 仕様確認、使い方相談、調査依頼 |

## 優先度ラベル

| ラベル | 用途 |
|--------|------|
| `priority:high` | 次のリリース候補に入れたい |
| `priority:medium` | 対応したいが、直近必須ではない |
| `priority:low` | backlog 管理用。すぐには着手しない |

## 参加しやすさラベル

| ラベル | 用途 | 付ける条件 |
|--------|------|------------|
| `good first issue` | 初回貢献向け | 変更範囲が狭く、セットアップ後に着手しやすい |
| `help wanted` | 外部協力歓迎 | 方針は決まっていて、実装や検証を依頼したい |

## 領域ラベル

| ラベル | 用途 |
|--------|------|
| `area:tactics` | 戦術シミュレーター関連 |
| `area:team-management` | チーム・選手管理関連 |
| `area:team-manual` | チームマニュアル関連 |
| `area:code-lab` | コードラボ関連 |
| `area:electron` | デスクトップ配布関連 |
| `area:ci` | CI / workflow / release 関連 |

## 推奨運用

1. すべての Issue に `bug` / `enhancement` / `documentation` / `question` のいずれかを付けます。
2. 優先順位を判断できる場合は `priority:*` を 1 つだけ付けます。
3. 対象範囲が明確なら `area:*` を 1 つ以上付けます。
4. 外部コントリビュータに開けるものだけ `good first issue` または `help wanted` を付けます。

## `good first issue` の基準

次の条件をすべて満たすものを推奨します。

- 変更対象ファイルが少ない
- 期待結果が README / docs / UI テキストで明確
- ドメイン知識が深くなくても進められる
- 既存テストの追加方針が分かりやすい

避けるべきもの:

- アーキテクチャ横断の変更
- Electron 配布やビルド周りの複雑な修正
- 戦術シミュレーターの深い仕様理解が必要なもの

## `help wanted` の基準

次のような Issue に付けます。

- 方針は固まっているが、実装量がある
- デザイン、文書、テスト追加などで協力を募りたい
- メンテナー単独だと後回しになりやすい

## triage の流れ

1. Issue を受けたら再現性とカテゴリを確認します。
2. 不足情報が多い場合は追加質問を返します。
3. ラベルを付けて優先順位を決めます。
4. すぐ着手しないものも close せず、backlog として残します。

## 参考

- bug と feature request の投稿フォームは `.github/ISSUE_TEMPLATE/` にあります。
- PR 側の確認項目は `.github/PULL_REQUEST_TEMPLATE.md` を使います。
